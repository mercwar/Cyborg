### 1. Language overview — Cyborg ev2.1 (Constellation Patch Language)

**Purpose:**  
Cyborg ev2.1 is the **patch language** Constellation uses to:

- Describe binary‑level intent (`MZ` + `ASM`)  
- Express structured logic (`C`)  
- Embed or reference LLM intelligence (`LLM HEX`)  
- Keep everything **hex‑visible**, **schema‑described**, and **robot‑safe**  

Think of it as:

> “A patch file that can talk to EXEs, C compilers, and LLMs at the same time.”

---

### 2. Layer 0 — `MZ` envelope

Every Cyborg ev2.1 patch file starts with an **MZ‑style header block**, but symbolic, not raw DOS:

```text
MZ-PATCH
TARGET: /path/to/binary.exe
MODE:   READ_ONLY | SIMULATE | APPLY
ARCH:   x86_64
```

**Rules:**

- `MZ-PATCH` is the magic string.  
- `TARGET` is required (file path or symbolic name).  
- `MODE` controls whether this is a dry run or real patch.  
- `ARCH` must match the target’s architecture.

Constellation uses this header to:

- Decide if the patch is allowed to run.  
- Route it to the right robot (Sentinel, Cyborg, etc.).  

---

### 3. Layer 1 — `ASM` (symbolic x86 patch ops)

Inside the patch, you define **symbolic assembly blocks**:

```asm
[ASM]
; patch entry point
ORG 0x401000

MOV EAX, 0x1
CALL 0x402000
JMP  0x403000
```

**Key points:**

- `ORG` is symbolic — Constellation can map it to real offsets.  
- Only a **safe subset** of x86 is allowed (no raw self‑modifying code, no privileged ops).  
- Every ASM block is **annotated** and **schema‑described** for robots.

You can also define **named patch regions**:

```asm
[ASM:HOOK_MAIN]
HOOK AT 0x401050
    PUSH EAX
    CALL 0x500000 ; Cyborg handler
    POP  EAX
ENDHOOK
```

---

### 4. Layer 2 — `C` (structured logic + ABI surface)

Cyborg ev2.1 lets you embed **C fragments** that represent the logic behind the patch.

```c
[C]
/* PATCH: main_hook
 * PURPOSE: Intercept main entry and log startup.
 */
__stdcall void cyborg_main_hook(void) {
    log_event("Program started");
}
```

**Rules:**

- C blocks are **not** compiled directly in the patch — they are **semantic descriptions**.  
- Constellation + Cyborg can map them to:  
  - existing functions,  
  - stub functions,  
  - or LLM‑generated code.  
- ABI must be explicit: `__stdcall`, `__fastcall`, etc.

You can bind ASM → C:

```text
[BIND]
ASM: HOOK_MAIN
C:   cyborg_main_hook
```

---

### 5. Layer 3 — `LLM HEX` (GGUF + LLAMA payloads)

This is the **intelligence layer**.

Cyborg ev2.1 can embed or reference **LLM payloads** in **hex‑safe form**, aligned with:

- **GGUF** (binary model format)  
- **LLAMA** (model family / runtime)  

Two modes:

#### 5.1 Embedded HEX payload

```text
[LLM-HEX]
FORMAT: GGUF
MODEL:  cyborg_patch_assistant
SIZE:   0x0001F400  ; 128 KB

DATA:
48656C6C6F2C20576F726C64210A...
...
ENDDATA
```

#### 5.2 Referenced model

```text
[LLM-HEX]
FORMAT: LLAMA
MODEL:  cyborg://models/patch/ev2.1/base
MODE:   REFERENCE
```

**Semantics:**

- `FORMAT` tells Constellation how to interpret the payload.  
- `MODEL` is either a symbolic name or a path.  
- `DATA` is hex‑encoded binary (GGUF chunk, prompt, adapter, etc.).  

This allows:

- Patches that **carry their own intelligence**.  
- Patches that **call into shared LLMs**.

### 6. Full Cyborg ev2.1 patch file example

```text
MZ-PATCH
TARGET: /Cyborg/engine/cyborg.exe
MODE:   SIMULATE
ARCH:   x86_64

[ASM:HOOK_MAIN]

    PUSH EAX
    CALL 0x500000 ; Cyborg handler
    POP  EAX

[C]
__stdcall void cyborg_main_hook(void) {
    // High-level description of what this hook does
    // (robots + LLMs can expand this into real code)
    log_event("Cyborg ev2.1 patch engaged");
}

[BIND]
ASM: HOOK_MAIN
C:   cyborg_main_hook

[LLM-HEX]
FORMAT: GGUF
MODEL:  cyborg://models/patch/ev2.1/assistant
MODE:   REFERENCE
```

### 7. How Constellation uses Cyborg ev2.1

When Constellation sees a Cyborg ev2.1 patch:

1. **MZ layer**  
   - Validates target, mode, arch.  

2. **ASM layer**  
   - Builds a symbolic patch graph.  
   - Validates safety.  

3. **C layer**  
   - Extracts semantic intent.  
   - Feeds it to LLMs if needed.  

4. **LLM HEX layer**  
   - Loads or references GGUF/LLAMA payloads.  
   - Uses them to generate or explain patch behavior.  

5. **AVIS / Datalake**  
   - Stores the patch as an **intelligence artifact**.  
   - Makes it searchable and explainable.
