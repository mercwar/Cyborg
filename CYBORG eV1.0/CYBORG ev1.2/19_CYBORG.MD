## **6.6 Overview**

The **Runtime Purity Enforcement** rules define how the Runtime Layer guarantees that:

- the C Layer remains pure  
- verb functions remain pure (except `build`)  
- no external contamination occurs  
- no side effects leak into or out of execution  
- no impurity is tolerated at any stage  

These rules are the **execution‚Äëtime counterpart** to the C Layer Purity Rules (Page 5.8).  
Where Page 5.8 defines what purity *is*, Page 6.6 defines how the runtime *enforces* it.

This section is **normative**.

---

# **6.6.1 Purity Enforcement Principle 1 ‚Äî No Side Effects Allowed**

During execution, the runtime MUST enforce that:

- no I/O occurs  
- no logging occurs  
- no printing occurs  
- no file access occurs  
- no network access occurs  
- no global state is modified  
- no memory is allocated or freed  

### **Meaning**  
If a verb attempts a side effect, the runtime MUST NOT:

- catch it  
- repair it  
- compensate for it  

The module simply fails.

---

# **6.6.2 Purity Enforcement Principle 2 ‚Äî No External Calls**

The runtime MUST prevent:

- system calls  
- library calls  
- host API calls  
- OS interactions  
- dynamic linking  
- plugin loading  

### **Meaning**  
Only verb functions may execute.

---

# **6.6.3 Purity Enforcement Principle 3 ‚Äî No Memory Mutation (Except Build)**

The runtime MUST enforce:

- `validate` ‚Üí read‚Äëonly  
- `scan` ‚Üí read‚Äëonly  
- `give` ‚Üí read‚Äëonly  
- `explain` ‚Üí read‚Äëonly

### **Meaning**  
Only `build` may modify context fields.

# **6.6.4 Purity Enforcement Principle 4 ‚Äî No Global State Access**

The runtime MUST prevent:

- reading global variables  
- writing global variables  
- referencing static state  
- referencing thread‚Äëlocal state  
- referencing environment variables  

### **Meaning**

### **Meaning**

### **Meaning**
