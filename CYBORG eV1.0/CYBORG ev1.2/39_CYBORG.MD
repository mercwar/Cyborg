## **9.2 Overview**

While Page 9.1 defined the **syntax** of the IR,  
Page 9.2 defines its **semantics** ‚Äî the rules that govern what each IR construct *means* and how it shapes the execution of a Cyborg Module.

IR semantics are:

- deterministic  
- declarative  
- non‚Äëinferential  
- non‚Äëprocedural  
- non‚Äëinteractive  
- non‚Äëcontextual  

The IR does not ‚Äúrun.‚Äù  
The IR **declares**.

The Runtime Layer **executes**.

This section is **normative**.

---

# **9.2.1 Semantic Rule 1 ‚Äî TARGET Defines Domain**

The `TARGET` declaration determines:

- which context type is allocated  
- which verb signatures are bound  
- which semantic domain the module operates in  

### **Meaning**
```
TARGET = AI
```
binds the module to **CyborgContext**.

```
TARGET = FILE
```
binds the module to **CyborgFile**.

```
TARGET = LAYER
```
binds the module to **CyborgLayer**.

The target is the **semantic universe** of the module.

---

# **9.2.2 Semantic Rule 2 ‚Äî OPEN_PAYLOAD Begins Execution Body**

`OPEN_PAYLOAD` marks the beginning of the verb sequence.

It does **not**:

- allocate memory  
- initialize context  
- perform any action  

It simply declares:

> ‚ÄúEverything after this line is executable semantic intent.‚Äù

---

# **9.2.3 Semantic Rule 3 ‚Äî SEAL Ends Execution Body**

`SEAL` marks the end of the verb sequence.

It does **not**:

- finalize context  
- free memory  
- perform cleanup  

It simply declares:

> ‚ÄúNo further verbs exist.‚Äù

---

# **9.2.4 Semantic Rule 4 ‚Äî Verb Lines Declare Execution Steps**

Each line of the form:

```
EXEC_<VERB>
```

declares:

- that the verb MUST be executed  
- in the order written  
- with the context bound by TARGET  
- with deterministic semantics  
- with no inference  

### **Meaning**
The IR is a **strict execution schedule**.

---

# **9.2.5 Semantic Rule 5 ‚Äî IR Does Not Contain Logic**

The IR MUST NOT contain:

- conditions  
- loops  
- branches  
- expressions  
- parameters  
- dynamic constructs  

### **Meaning**
IR is **not a programming language**.  
It is a **semantic declaration language**.

---

# **9.2.6 Semantic Rule 6 ‚Äî IR Does Not Modify Semantics of Verbs**

The IR determines:

- *which* verbs run  
- *in what order*  

The IR does **not** determine:

- *how* verbs behave  
- *what* verbs mean  
- *what* verbs are allowed to do  

### **Meaning**
Verb semantics are **fixed and immutable**.

---

# **9.2.7 Semantic Rule 7 ‚Äî IR Does Not Modify Context**

The IR does **not**:

- allocate context  
- initialize context  
- mutate context  
- destroy context  

### **Meaning**
Context lifecycle is governed by the **Runtime Layer**, not IR.

---

# **9.2.8 Semantic Rule 8 ‚Äî IR Is Deterministic**

Given the same IR:

- the same C code MUST be generated  
- the same verbs MUST be invoked  
- in the same order  
- with the same aggregation semantics  

### **Meaning**
IR is **mathematically reproducible**.

---

# **9.2.9 Semantic Rule 9 ‚Äî IR Is Non‚ÄëInferential**

The IR MUST NOT:

- imply missing verbs  
- imply missing structure  
- imply missing semantics  
- imply missing behavior  

### **Forbidden**
If the IR omits VALIDATE, the runtime MUST NOT infer it.

If the IR omits BUILD, the runtime MUST NOT infer it.

### **Meaning**
IR is **explicit**, never implicit.

---

# **9.2.10 Semantic Rule 10 ‚Äî IR Is Closed**

The IR MUST NOT:

- define new verbs  
- define new targets  
- define new constructs  
- extend the language  

### **Meaning**
The IR is a **closed semantic system**.

---

### **Semantic Interpretation**

- a CyborgContext is allocated  
- verbs run in the order written  
- no inference occurs  
- no optimization occurs  
- no skipping occurs  
- no branching occurs  
- no dynamic behavior occurs  

This is the **canonical IR semantic flow**.
